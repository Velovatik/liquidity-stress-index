# Индекс ликвидностного стресса (Liquidity Stress Index, LSI) — демо для хакатона

Небольшая демонстрация на **Go + Postgres**, показывающая базовый путь данных:

1. **Симулированные дневные метрики** приходят в виде CSV-снимка.
2. За один проход **ETL** читает файл, нормализует индикаторы (мин–макс), складывает их в взвешенный **индекс стресса** по шкале 0–100 и пишет всё в Postgres.
3. **REST API** (стандартный `net/http`) отдаёт `/health`, `/lsi/latest`, `/lsi/history`.

Никаких очередей, Kafka, ML, веб-фреймворков — только понятность для мастер-класса минут на 30.

## Как устроен индекс

Для каждого индикатора считаем:

- `вклад_стресса = (значение - min(колонка)) / (max(колонка) - min(колонка))`

**Банковские резервы** и **баланс казначейства** переворачиваем (ниже буферы → выше стресс):

- `вклад_стресса = 1 - масштабированное сырое`

Взвешенная сумма:

- `LSI = 100 * Σᵢ (0.2 * вклад_стресса_i)` по пяти индикаторам.

Коэффициенты в сумме дают 1, поэтому взвешенная сумма лежит в **[0, 1]** до умножения на 100.

## Запуск через Docker Compose

Нужно: Docker Desktop или эквивалент Compose v2.

Из корня репозитория:

```bash
docker-compose up --build
# В Compose v2: `docker compose up --build`.
```

Контейнер приложения **ждёт Postgres**, применяет `migrations/init.sql`, выполняет **ETL один раз**, затем поднимает HTTP на **`http://localhost:8080`**.

### Быстрая проверка

```bash
curl -s http://localhost:8080/health
curl -s http://localhost:8080/lsi/latest | jq
curl -s http://localhost:8080/lsi/history | jq
```

`GET /lsi/latest` отвечает `{ "lsi": <float>, "date": "YYYY-MM-DD" }`.  
`GET /lsi/history` — JSON-массив таких объектов за последние **30** дней (по возрастанию даты).

## Структура проекта

```
cmd/app/main.go            # Запуск: ожидание БД → миграции → ETL → API
internal/etl/pipeline.go    # CSV, нормализация, вставка в БД
internal/etl/sample_csv.go # Создание data/liquidity.csv, если файла нет
internal/api/router.go     # net/http маршруты и обработчики
internal/storage/*.go       # доступ к Postgres через pgx pool
data/liquidity.csv         # Синтетическая история (~100 строк данных)
migrations/init.sql       # таблица и индекс
Dockerfile / docker-compose.yml
```

При каждом старте логически **очищаем** таблицу (`DELETE FROM lsi_data`) перед повторной загрузкой CSV, чтобы можно было снова поднять Compose без ручной чистки.

Если `data/liquidity.csv` отсутствует, приложение восстанавливает детерминированный файл при старте через `EnsureLiquidityCSV`.

## Разработка без Docker

Postgres должен слушать `localhost:5432`, БД `lsi`, пользователь/пароль `lsi`.

```bash
export DATABASE_URL='postgres://lsi:lsi@localhost:5432/lsi?sslmode=disable'
go run ./cmd/app
```

Можно оставить только Postgres из Compose: `docker compose up postgres`.

## Ограничения — намеренно

- ETL последовательный, один экземпляр API — удобнее рассказывать, это не промышленные SLA.
- Нормализация только по окну данных в CSV; в продакшене окна задают отдельно.
- Нет аутентификации и авторизации — держите демо в изолированной сети.
